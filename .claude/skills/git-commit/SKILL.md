# Git Commit Creator

작업 브랜치의 변경 사항을 `.gitmessage.txt` 컨벤션에 맞춰 커밋합니다.

## 커밋 형식 (.gitmessage.txt)

```
#{이슈번호} {타입}({범위}): {제목}   ← 제목 40자 이내

{본문}                              ← 선택, 한 줄 띄우고 72자 줄바꿈

Resolves: #{이슈번호}
See also: None
```

- **타입**: `feat` `fix` `docs` `style` `refactor` `test` `chore`
- **범위**: `ui` `state` `route` `api` `auth` `domain` `db` `infra` `config` `deps` `security` `i18n` `docs` `test` `etc` (해당 시, 애매하면 생략)
- **Co-Authored-By는 붙이지 않습니다.**

## Instructions

1. **이슈번호 확인**: 사용자에게 이 작업의 이슈번호를 물어봅니다.
   ```bash
   git rev-parse --abbrev-ref HEAD
   ```
   - 브랜치명 형식은 `{기능종류}/{설명}` (예: `feat/login-oauth-api`).
   - 브랜치의 기능종류(`feat`/`fix`/…)는 커밋 타입 추론의 **참고용**일 뿐, 실제 타입은 변경 내용으로 판단합니다.
   - 이슈번호는 브랜치명에 없으므로 사용자에게 확인합니다. 없으면 `Resolves` 줄을 생략합니다.

2. **변경 분석 및 그룹핑**:
   - `git status`와 `git diff`(스테이징 전후 모두)로 변경 파일을 확인합니다.
   - 변경 내용을 분석해 **비슷한 작업끼리 그룹핑**하고, 각 그룹의 타입·범위를 추론합니다.

3. **커밋 계획 보고**: 아래를 보고하고 **사용자 승인을 받습니다**.
   - **🌿 브랜치 / 이슈**: `{브랜치명}` → `#{이슈번호}`
   - **📊 변경 사항**: 변경 파일 목록
   - **🎯 커밋 계획**:
     | 그룹 | 파일 | 커밋 제목 (`#N type(scope): ...`) | 본문 |
     |------|------|------|------|

4. **커밋 실행**: 승인 후에만 그룹별로 순차 실행합니다.
   ```bash
   git add {files}
   git commit -m "#{번호} {타입}({범위}): {제목}" -m "{본문}" -m "Resolves: #{번호}
   See also: None"
   ```
   - 제목 40자, 본문 72자 이내를 지킵니다.
   - 범위가 애매하면 `{타입}: {제목}`처럼 범위를 생략합니다.

5. **푸시 확인**: 모든 커밋 완료 후 푸시 여부를 물어봅니다.

## 중요 제약 사항

- **승인 후 커밋**: 사용자가 명시적으로 승인한 경우에만 커밋을 실행합니다.
- **자동 커밋/푸시 금지**: 승인 없이 먼저 커밋하거나 푸시하지 않습니다.
- **매칭 실패 시**: 이슈번호를 확정할 수 없으면 억지로 넣지 말고 사용자에게 확인합니다.
